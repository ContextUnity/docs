ContextUnity Security Scope

ContextUnity uses stateless cryptographic tokens (ContextToken) to handle multi-tenancy and authentication without the need for central database lookups. The ecosystem supports two distinct signing approaches, balancing simplicity for internal networks against strict zero-trust security for enterprise deployments.

1. Symmetric Approach (HMAC) — Internal Ecosystem

When all services belong to one organization and reside within a trusted closed network (VPC), the simplest approach is pure symmetric signing.

This model is built-in to the open-source contextcore package and does not require third-party key managers.

How It Works

  1. Shared Secret: A single SIGNING_SHARED_SECRET is generated.
  2. Distribution: This secret is securely shared via .env files to all trusted services in the network (e.g., the Store backend and the Central Router).
  3. Signing: When the Store backend needs to communicate with the Router to query the ContextBrain, it generates a ContextToken dynamically for its tenant (tenant_id="store") and hashes it using the shared secret.
  4. Verification: The Router receives the token and re-hashes it. If the hash matches, the token is perfectly valid and trusted.

The Trust Model (Internal Service Mesh)

In this approach, there is no central registry of “which tenant belongs to which service.” The Router assumes that any service possessing the SIGNING_SHARED_SECRET is fully trusted.

Risks: Because the Store backend possesses the same master secret as the Router, there is a theoretical risk. Should the Store backend be compromised or its developers act maliciously, they could forge a token with a different tenant ID (e.g., tenant_id="medical_records") and sign it. The Router will accept this token because the cryptographic signature is mathematically valid. Thus, this model operates on an “all in the family” trust basis.

2. Asymmetric Approach (Ed25519 / Cloud KMS) — Zero-Trust

For enterprise scenarios where multiple distinct organizations, third parties, or isolated microservices connect to one unified Brain & Router, the symmetric model isn’t secure enough.

This is where the ContextShield package provides asymmetric backing. It guarantees that a compromised service can never forge tokens for other tenants, as validation happens via a registry of public keys.

ContextShield provides two asymmetric backends:

  1. Ed25519Backend: Fast, local asymmetric curves.
  2. KmsBackend: Hardware-backed Enterprise signing via Cloud Key Management Services (e.g., Google Cloud KMS).

How It Works (Example: KmsBackend)

  1. Hardware Keys: A cryptographic Key Pair is generated directly inside Google Cloud KMS (the Private Key never leaves the Hardware Security Module - HSM).
  2. Key Designation (kid): The Router (or verifier) is configured to trust the KMS Public Key, identifying it via a Key ID (kid, e.g., kms-001).
  3. Signing:
    • When a service needs to generate a token, it delegates the singing operation to the Cloud KMS API.
    • ContextShield sends the base64-encoded token payload to Google Cloud via the google-cloud-kms client.
    • KMS returns a cryptographic signature.
    • The final format becomes: kid.payload_b64.signature_b64.
  4. Verification & Authorization:
    • The receiving Router looks at the kid and retrieves the corresponding Public Key from the cloud provider (caching it locally to avoid round-trips).
    • It verifies the ECDSA(SHA256) math using the cryptography library locally.
    • If the signature is valid, the Router trusts the token’s payload (e.g., the tenant_id requested).

The Trust Model (Zero-Trust)

Because the Private Key never exists within the memory of your application servers, even a total compromise of the service container won’t leak the key. Furthermore, the Router can map specific kids to explicitly allowed tenants. This guarantees total zero-trust isolation, preventing horizontal privilege escalation across your monorepo or service mesh.

3. Transport Security (gRPC TLS)

Cryptographic token signing guarantees authenticity, but the data itself must be protected while traveling over the network. ContextUnity services communicate primarily via high-performance gRPC, which mandates encryption.

Standard TLS

By default, when GRPC_TLS_ENABLED=true, ContextUnity services enforce TLS server verification.

  • The server initializes using GRPC_TLS_SERVER_CERT and GRPC_TLS_SERVER_KEY.
  • The client connects using GRPC_TLS_CA_CERT to verify it’s talking to the legitimate Router/Brain.
  • The client does not prove its identity at the TCP level; its identity is proven strictly via the ContextToken interceptors.

This standard model protects against Man-in-the-Middle (MitM) attacks.

Mutual TLS (mTLS) — Zero-Trust (ContextShield)

For strict Zero-Trust deployments (like HIPAA compliance), ContextShield provides helpers for Mutual TLS (mTLS). In this mode, the client is additionally required to present a physical X.509 certificate to establish the TCP connection.

mTLS is enabled automatically when the client presents:

  • GRPC_TLS_CLIENT_CERT
  • GRPC_TLS_CLIENT_KEY

When both sides require mTLS, the Router refuses connection handshakes from any worker or client that lacks a valid certificate signed by the internal CA. Even if an attacker steals a perfectly valid, correctly signed ContextToken, their network request is dropped before it reaches the application layer, adding a powerful layer of defense-in-depth networking.

4. Application Secrets & Vault (ContextShield)

Beyond tokens and network encryption, applications (like Python agents in ContextRouter) frequently need API keys, database credentials, or third-party service tokens to perform their tasks.

Rather than injecting sensitive plaintext passwords directly into agents, ContextShield provides a dedicated gRPC Secrets API acting as a central Vault.

Access Control

To interact with the Secrets API, a service must present a ContextToken carrying specific permissions:

  • shield:secrets:read — required to read (GetSecret, ListSecrets, Decrypt)
  • shield:secrets:write — required to modify (PutSecret, RotateSecret)

Because tokens are securely signed (via KmsBackend or HmacBackend) and strictly scoped to a tenant_id, the Secrets API guarantees that Tenant A can never read Tenant B’s API keys.

How Shield Manages Secrets

ContextShield abstracts the physical storage of secrets through different backends:

  1. EnvSecretStore (Open-Source Default): Secrets are loaded securely into memory from environmental variables (e.g., SHIELD_SECRET_OPENAI_KEY). Good for basic setups.
  2. PgSecretStore / VaultSecretStore (Enterprise): Real-time, encrypted-at-rest vault solutions leveraging PostgreSQL or HashiCorp Vault.

Agent Workflow (Example)

When a ContextRouter Agent needs to call OpenAI:

  1. The Agent does not contain sk-proj-... in its code.
  2. Instead, the Agent makes a gRPC call to ContextShield asking for openai_api_key.
  3. Shield verifies the agent’s ContextToken, checks its tenant_id, and authenticates the read permission.
  4. Shield retrieves the secret from its active backend (e.g., HashiCorp Vault) and returns it over the mTLS encrypted channel.
  5. The Agent uses the key securely in memory and discards it after the execution cycle.

This ensures credentials remain centrally rotated, audited, and tightly segregated per project tenant.

5. Database-Level Isolation (PostgreSQL RLS)

Even if an attacker bypasses the Application Layer or a developer writes a vulnerable SQL tool (SQL Injection / Prompt Injection), ContextUnity enforces a final “Fail-Closed” layer of defense inside the database kernel.

ContextBrain and ContextCommerce utilize PostgreSQL Row-Level Security (RLS) to enforce strict Role-Based Access Control (RBAC) across two dimensions.

Dual-Dimensional RLS Policies

When a gRPC handler connects to the PostgreSQL pool, it passes the verified ContextToken. The database connection wrapper performs the following before executing any application query:

All tables containing personal or project data (e.g., knowledge_nodes, episodic_events, user_facts) carry hardened RLS policies:

  1. Tenant Isolation (B2B):

    Guarantees that Project A can never SELECT or UPDATE rows belonging to Project B.

  2. User Isolation (B2C):

    Guarantees that inside the same Tenant, User A can never see User B’s episodic memory or personal RAG vectors.

Role Enforcement

The application itself connects using the brain_app PostgreSQL role. This role does not have the BYPASSRLS privilege, meaning the PostgreSQL kernel rigidly enforces these policies on every query. The brain_admin role (used exclusively by the backend dashboard) is the only role capable of viewing cross-tenant metrics.
